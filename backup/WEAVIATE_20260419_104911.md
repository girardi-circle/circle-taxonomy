# WEAVIATE.md

## Weaviate Collections & Query Patterns

Weaviate serves three purposes in this system:
1. **Classification matching** — finding existing subtopics that match new issues (SubTopic collection)
2. **Issue-level RAG** — retrieving relevant issues for conversational intelligence (ClassifiedIssue collection)
3. **Conversation-level RAG** — retrieving full transcripts for account history, similar calls, and agent coaching (Transcript collection)

Weaviate does not store API keys, passwords, or any data that doesn't also exist in Redshift. Redshift is the source of truth.

---

## Collection 1: SubTopic

**Purpose:** Classification pipeline (Step 2). Small, high-precision collection for matching new issues against existing subtopics.

### Properties

| Property | Type | Vectorized | Description |
|----------|------|-----------|-------------|
| subtopic_id | int | No | FK to `taxonomy.sub_topics.id` |
| topic_id | int | No | FK to `taxonomy.topics.id`. Enables filtered queries. |
| product_area_id | int | No | FK to `taxonomy.product_areas.id`. Nullable. Enables filtered queries. |
| name | text | No | Subtopic name. Stored for readability, not vectorized. |
| canonical_description | text | **Yes** | Primary vector field. Searched against issue `segment_description`. |

**Expected size:** 50-500 records. Grows as new subtopics are approved.

**Populated by:** Step 3 (when emerging candidates are approved). Updated by Step 4 (centroid maintenance).

---

## Collection 2: ClassifiedIssue

**Purpose:** RAG chat for issue-level intelligence. When a user asks "why are customers frustrated with the editor?", this collection finds the most semantically relevant issues within the user's active filters.

### Properties

| Property | Type | Vectorized | Description |
|----------|------|-----------|-------------|
| issue_id | int | No | FK to `taxonomy.classified_issues.id` |
| transcript_id | int | No | FK to `taxonomy.transcripts.id`. Links issue to source conversation. |
| sub_topic_id | int | No | FK to `taxonomy.sub_topics.id`. Nullable (unmatched issues). Enables filtered queries. |
| topic_id | int | No | FK to `taxonomy.topics.id`. Nullable. Enables filtered queries. |
| product_area_id | int | No | FK to `taxonomy.product_areas.id`. Nullable. Enables filtered queries. |
| nature_id | int | No | FK to `taxonomy.natures.id`. Enables filtered queries. |
| intent_id | int | No | FK to `taxonomy.intents.id`. Enables filtered queries. |
| sentiment | text | No | positive, negative, neutral, frustrated. Enables filtered queries. |
| classified_at | date | No | Enables timeframe filtering. |
| source_url | text | No | Link to original ticket/call. Passed to UI for source attribution. |
| segment_description | text | **Yes** | Primary vector field. Searched against user's chat question. |
| verbatim_excerpt | text | No | Raw customer words. Sent to Claude as context (avoids Redshift round trip). |

**Expected size:** 13k+ records, growing with each extraction run.

**Populated by:** Phase 2a vectorization pipeline. Incrementally updated after each Step 1 (extraction) and Step 2 (classification) run.

---

## Collection 3: Transcript

**Purpose:** RAG chat for conversation-level intelligence. Enables "find similar conversations", account history, agent coaching, and competitive intelligence.

### Properties

| Property | Type | Vectorized | Description |
|----------|------|-----------|-------------|
| transcript_id | int | No | FK to `taxonomy.transcripts.id` |
| source_id | text | No | Original ticket/call ID. |
| source_type | text | No | 'zendesk', 'fathom', 'slack'. Enables filtered queries. |
| community_id | int | No | Circle community ID. Enables account-level filtering. |
| title | text | No | Ticket subject or call title. Readability. |
| source_url | text | No | Link to original ticket/call. Source attribution. |
| summary | text | No | Claude-generated summary. Sent as context to avoid re-reading raw_text. |
| raw_text | text | **Yes** | Primary vector field. Full conversation vectorized for semantic search. |

**Expected size:** 13k+ records, growing with each ingestion run.

**Populated by:** Phase 2a vectorization pipeline. Incrementally updated after each Step 1 (extraction) run.

## Vectorizer configuration

All three collections use the same embedding model. Whichever model is chosen, it must be used consistently across all collections and all query-time embeddings.

**Options under consideration:**
- `text2vec-openai` (text-embedding-3-small)
- `text2vec-cohere` (embed-english-v3.0)
- Voyage AI embeddings
- Weaviate's built-in `text2vec-transformers`

### Index configuration

```python
{
    "vectorIndexType": "hnsw",
    "vectorIndexConfig": {
        "ef": 128,
        "efConstruction": 256,
        "maxConnections": 32
    }
}
```

## Query patterns by collection

### SubTopic queries (classification pipeline)

**Unfiltered top-N search** (Step 2.2):
```python
result = (
    client.query
    .get("SubTopic", ["subtopic_id", "topic_id", "product_area_id", "name", "canonical_description"])
    .with_near_text({"concepts": [segment_description]})
    .with_limit(5)
    .with_additional(["distance"])
    .do()
)
```

**Filtered by product area** (when product area is known):
```python
result = (
    client.query
    .get("SubTopic", ["subtopic_id", "topic_id", "name", "canonical_description"])
    .with_near_text({"concepts": [segment_description]})
    .with_where({
        "path": ["product_area_id"],
        "operator": "Equal",
        "valueInt": known_product_area_id
    })
    .with_limit(5)
    .with_additional(["distance"])
    .do()
)
```

**Second-pass unfiltered check** (Step 2.4 — prevents cross-product-area duplicates):
```python
result = (
    client.query
    .get("SubTopic", ["subtopic_id", "topic_id", "product_area_id", "name", "canonical_description"])
    .with_near_text({"concepts": [segment_description]})
    .with_limit(3)
    .with_additional(["distance"])
    .do()
)
# Only consider results with distance < 0.20
```

**Duplicate detection** (Step 4):
```python
for subtopic in all_subtopics:
    result = (
        client.query
        .get("SubTopic", ["subtopic_id", "name"])
        .with_near_text({"concepts": [subtopic["canonical_description"]]})
        .with_where({
            "path": ["subtopic_id"],
            "operator": "NotEqual",
            "valueInt": subtopic["subtopic_id"]
        })
        .with_limit(3)
        .with_additional(["distance"])
        .do()
    )
    # Flag pairs with distance < DUPLICATE_DETECTION_THRESHOLD
```

### ClassifiedIssue queries (RAG chat)

**Filtered semantic search** (primary RAG retrieval):
```python
# Build where filter from active UI filters
where_filter = {
    "operator": "And",
    "operands": [
        {"path": ["product_area_id"], "operator": "Equal", "valueInt": 1},
        {"path": ["sentiment"], "operator": "Equal", "valueText": "frustrated"},
        {"path": ["classified_at"], "operator": "GreaterThan", "valueDate": "2026-01-01T00:00:00Z"},
        {"path": ["classified_at"], "operator": "LessThan", "valueDate": "2026-04-01T00:00:00Z"}
    ]
}

result = (
    client.query
    .get("ClassifiedIssue", [
        "issue_id", "transcript_id", "sub_topic_id", "nature_id", "intent_id",
        "sentiment", "segment_description", "verbatim_excerpt", "source_url"
    ])
    .with_near_text({"concepts": [user_question]})
    .with_where(where_filter)
    .with_limit(20)  # RAG_ISSUE_RETRIEVAL_LIMIT
    .with_additional(["distance"])
    .do()
)

# Discard results above RAG_RELEVANCE_THRESHOLD
results = [r for r in result if r["_additional"]["distance"] < 0.40]
```

**Unfiltered semantic search** (when user hasn't applied filters):
```python
result = (
    client.query
    .get("ClassifiedIssue", [
        "issue_id", "transcript_id", "segment_description", "verbatim_excerpt",
        "sentiment", "source_url"
    ])
    .with_near_text({"concepts": [user_question]})
    .with_limit(20)
    .with_additional(["distance"])
    .do()
)
```

### Transcript queries (RAG chat — conversation-level)

**Account history** (all conversations for a community):
```python
result = (
    client.query
    .get("Transcript", [
        "transcript_id", "source_id", "source_type", "title",
        "summary", "source_url"
    ])
    .with_near_text({"concepts": [user_question]})
    .with_where({
        "path": ["community_id"],
        "operator": "Equal",
        "valueInt": community_id
    })
    .with_limit(10)  # RAG_TRANSCRIPT_RETRIEVAL_LIMIT
    .with_additional(["distance"])
    .do()
)
```

**Find similar conversations** (semantic search across all transcripts):
```python
result = (
    client.query
    .get("Transcript", [
        "transcript_id", "source_id", "source_type", "title",
        "summary", "source_url"
    ])
    .with_near_text({"concepts": [reference_transcript_summary]})
    .with_where({
        "path": ["transcript_id"],
        "operator": "NotEqual",
        "valueInt": exclude_self_id
    })
    .with_limit(10)
    .with_additional(["distance"])
    .do()
)
```

**Filtered by source type and timeframe** (e.g., only Fathom calls in Q1):
```python
where_filter = {
    "operator": "And",
    "operands": [
        {"path": ["source_type"], "operator": "Equal", "valueText": "fathom"},
        # Note: Transcript collection would need a date field for timeframe filtering
        # Consider adding ingested_at or a created_at property
    ]
}

result = (
    client.query
    .get("Transcript", [
        "transcript_id", "title", "summary", "source_url"
    ])
    .with_near_text({"concepts": [user_question]})
    .with_where(where_filter)
    .with_limit(10)
    .with_additional(["distance"])
    .do()
)
```

## Write patterns

### Insert new subtopic (after approval — Step 3)
```python
client.data_object.create(
    class_name="SubTopic",
    data_object={
        "subtopic_id": new_subtopic_id,
        "topic_id": topic_id,
        "product_area_id": product_area_id,
        "name": subtopic_name,
        "canonical_description": canonical_description
    }
)
```

```python
client.data_object.create(
    class_name="ClassifiedIssue",
    data_object={
        "issue_id": issue_id,
        "transcript_id": transcript_id,
        "sub_topic_id": sub_topic_id,       # Can be None
        "topic_id": topic_id,               # Can be None
        "product_area_id": product_area_id, # Can be None
        "nature_id": nature_id,
        "intent_id": intent_id,
        "sentiment": sentiment,
        "classified_at": classified_at_iso,
        "source_url": source_url,
        "segment_description": segment_description,
        "verbatim_excerpt": verbatim_excerpt
    }
)
```

### Insert transcript (after ingestion)
```python
client.data_object.create(
    class_name="Transcript",
    data_object={
        "transcript_id": transcript_id,
        "source_id": source_id,
        "source_type": source_type,
        "community_id": community_id,
        "title": title,
        "source_url": source_url,
        "summary": summary,
        "raw_text": raw_text

```

### Update subtopic description (centroid maintenance — Step 4)
```python
result = (
    client.query
    .get("SubTopic", ["_additional { id }"])
    .with_where({"path": ["subtopic_id"], "operator": "Equal", "valueInt": target_id})
    .do()

```

result = (

    .get("ClassifiedIssue", ["_additional { id }"])

## Sync strategy: Redshift → Weaviate

Redshift is the source of truth. All three Weaviate collections are derived search indexes.

**Writes always go to Redshift first, then sync to Weaviate.** Never write to Weaviate without a corresponding Redshift record.

**Sync triggers:**
- Step 1 (extraction) completes → insert new issues into `ClassifiedIssue`, new transcripts into `Transcript`
- Step 2 (classification) completes → update `ClassifiedIssue` records with assigned subtopic metadata
- Step 3 (approval) completes → insert new subtopic into `SubTopic`, update linked `ClassifiedIssue` records
- Step 4 (maintenance) completes → update `SubTopic` descriptions

**Bulk sync** (`POST /api/pipeline/vectorize`):
- One-time operation for initial population
- Reads all records from Redshift, inserts into Weaviate
- Can be re-run to rebuild collections from scratch if drift is detected

**Reconciliation** (run during Step 4):
- Compare record counts: Weaviate collection size vs Redshift `COUNT(*)`
- Flag any discrepancies for investigation
- If Weaviate has orphaned records (IDs not in Redshift), delete them

## Distance interpretation

Weaviate returns cosine distance (0 = identical, 2 = opposite).

### For SubTopic matching (classification)

| Distance | Interpretation | Action |
|----------|---------------|--------|
| < 0.15 | Strong match | Band A: auto-assign |
| 0.15 – 0.35 | Ambiguous | Band B: Claude arbitration |
| > 0.35 | No match | Band C: propose new subtopic |

### For RAG retrieval (chat)

| Distance | Interpretation | Action |
|----------|---------------|--------|
| < 0.25 | Highly relevant | Include in Claude context, show as primary source |
| 0.25 – 0.40 | Somewhat relevant | Include in Claude context if under retrieval limit |
| > 0.40 | Not relevant | Discard, do not send to Claude |

RAG thresholds are configured in `shared/config.py` as `RAG_RELEVANCE_THRESHOLD`.

**All thresholds depend on the embedding model and need calibration with real data.**
