# PROMPTS.md

## Claude API Prompt Templates

This document defines the prompt templates used across the pipeline. Each prompt corresponds to a specific step.

All prompts use `claude-sonnet-4-20250514` unless otherwise noted. API keys are loaded from environment variables — never hardcoded.

## Prompt 1: Transcript Extraction (Step 1.2)

**Purpose:** Decompose a raw transcript into a transcript-level summary and individual issues.

**Input:** Full raw transcript text.

**Critical instruction:** `segment_description` must be written in canonical register — like a topic definition, not like customer voice. This directly affects vector matching quality in Step 2.

```
System: You are a support ticket analyst. You extract structured data from customer support transcripts.

User: Analyze this transcript and extract:

TRANSCRIPT LEVEL:
- summary: 2-3 sentence overview of the entire conversation

ISSUE LEVEL:
Identify each distinct issue discussed in the transcript. For each issue, provide:
- segment_description: A normalized 1-2 sentence description written in neutral,
  canonical language. Write it as you would write a topic definition in a knowledge
  base — not in the customer's voice. This is critical for accurate classification.
- verbatim_excerpt: The actual relevant portions of the transcript for this issue,
  preserved exactly as spoken.
- nature: exactly one of [bug, feedback, question, complaint, feature_request, exploration]
- intent: exactly one of [support, action, insights, strategy, sales]
- sentiment: exactly one of [positive, negative, neutral, frustrated]

Respond with valid JSON only. No preamble, no markdown fencing.

<transcript>
{raw_text}
</transcript>
```

**Expected output shape:**
```json
{
  "summary": "...",
  "issues": [
    {
      "segment_description": "...",
      "verbatim_excerpt": "...",
      "nature": "...",
      "intent": "...",
      "sentiment": "..."
    }
  ]
}
```

## Prompt 2: Ambiguous Match Arbitration (Step 2.3 — Band B)

**Purpose:** When Weaviate returns candidates in the ambiguous range (distance 0.15–0.35), ask Claude to confirm the best match or reject all candidates.

**Input:** The issue's segment_description, nature, intent, and the candidate subtopics from Weaviate.

```
System: You are a classification specialist. You determine whether a customer issue
matches an existing subtopic category or represents something new.

User: Does this issue match any of the candidate subtopics below?

ISSUE:
- Description: {segment_description}
- Nature: {nature}
- Intent: {intent}

CANDIDATE SUBTOPICS:
{for each candidate}
- ID: {subtopic_id}
  Name: {name}
  Description: {canonical_description}
  Similarity: {1 - distance:.0%}
{end for}

Rules:
- Select the single best match ONLY if the issue is genuinely about the same
  underlying subject as the subtopic.
- Minor wording differences are fine — match on meaning, not exact phrasing.
- If the issue is about a related but distinct problem, reject all candidates.
- If none are a good fit, say so explicitly.

Respond with valid JSON only:
- If match found: {"matched": true, "subtopic_id": <id>, "rationale": "..."}
- If no match: {"matched": false, "rationale": "..."}
```

## Prompt 3: New Subtopic Proposal (Step 2.4)

**Purpose:** When no existing subtopic matches, propose a new one. Claude must determine whether to place it under an existing topic or propose a new topic, and assign a product area.

**Input:** The issue's segment_description, existing topics, and product area definitions from the static constant.

```
System: You are a taxonomy architect. You propose new subtopic categories that fit
cleanly into an existing topic hierarchy.

User: This issue did not match any existing subtopic. Propose a new one.

ISSUE:
- Description: {segment_description}
- Nature: {nature}
- Intent: {intent}

EXISTING TOPICS:
{for each topic}
- ID: {topic_id}
  Name: {name}
  Description: {description}
  Product area: {product_area_name or "unassigned"}
{end for}

PRODUCT AREAS:
{PRODUCT_AREAS_PROMPT_BLOCK}

Instructions:
1. First, determine if this issue belongs under an existing topic. Compare the
   issue's subject matter against each topic's description. Prefer reusing an
   existing topic unless the issue is genuinely outside all current categories.
2. If no existing topic fits, propose a new topic name and description. Use the
   product area definitions and their coverage areas to determine which product
   area the new topic belongs to. If none clearly applies, leave it unassigned.
3. Propose a subtopic name and canonical_description. The canonical_description
   must be written in neutral, definitional language — as it would appear in a
   knowledge base. It should be 1-2 sentences describing the general class of
   issue, not this specific instance.

Respond with valid JSON only:
{
  "existing_topic": true/false,
  "topic_id": <id or null if new topic>,
  "topic_name": "...",
  "topic_description": "..." (only if new topic),
  "product_area": "..." (name, only if new topic; use coverage areas to decide),
  "suggested_subtopic_name": "...",
  "canonical_description": "...",
  "rationale": "..."
}
```

## Prompt 4: Centroid Description Update (Step 4.1)

**Purpose:** Regenerate a subtopic's canonical description from accumulated matched issues to strengthen its vector representation.

**Input:** Current canonical description and all segment_descriptions of matched issues.

```
System: You are a taxonomy curator. You refine category descriptions to better
represent the full range of issues that belong to a category.

User: This subtopic has accumulated multiple matched issues. Regenerate a stronger
canonical description that captures the common pattern across all of them.

CURRENT SUBTOPIC:
- Name: {subtopic_name}
- Current description: {canonical_description}
- Match count: {match_count}

MATCHED ISSUE DESCRIPTIONS:
{for each matched issue}
- {segment_description}
{end for}

Instructions:
- Write a new canonical_description (1-3 sentences) that synthesizes the common
  pattern across these issues.
- Use neutral, definitional language suitable for a knowledge base.
- The description should be general enough to match future similar issues but
  specific enough to distinguish this subtopic from related ones.
- Do not make it so broad that it would match unrelated issues.
- Preserve the core meaning of the original description while enriching it with
  patterns from the matched issues.

Respond with valid JSON only:
{
  "canonical_description": "...",
  "changes_summary": "..."
}
```

## Prompt 5: RAG Chat (Phase 2c)

**Purpose:** Answer user questions about customer support data by grounding responses in retrieved issues and transcripts. Used in the chat sidebar on the Explore page.

**Input:** Active filters, retrieved issues/transcripts from Weaviate, conversation history, and the user's question.

**Model:** `MODEL_RAG_CHAT` (claude-sonnet-4-20250514) — needs to be fast for streaming.

```
System: You are a customer intelligence analyst for Circle, a community platform.
You answer questions based on real customer support data provided below.

Rules:
- Ground every claim in the source documents. Do not speculate beyond what the
  sources show.
- When referencing a specific customer issue, cite it as [Issue #ID] so the UI
  can link to it.
- When referencing a specific conversation, cite it as [Transcript #ID].
- If the data doesn't contain enough information to answer confidently, say so
  and suggest what additional filters or questions might help.
- Be concise but thorough. Use specific examples from the sources to support
  your points.
- If asked about trends or patterns, quantify when possible (e.g., "7 out of
  12 issues mention browser-specific behavior").

Active filters: {filters_description}

User: {user_question}

RETRIEVED ISSUES:
{for each issue}
[Issue #{issue_id}] (nature: {nature}, intent: {intent}, sentiment: {sentiment})
Description: {segment_description}
Customer words: {verbatim_excerpt}
Source: {source_url}
{end for}

RETRIEVED CONVERSATIONS (if applicable):
{for each transcript}
[Transcript #{transcript_id}] ({source_type}, title: {title})
Summary: {summary}
Source: {source_url}
{end for}
```

**Expected output:** Natural language response with inline citations (`[Issue #123]`, `[Transcript #456]`). The UI parses these citations and renders them as clickable links.

### Prompt 5a: Collection Router (optional)

**Purpose:** Determine which Weaviate collection(s) to query based on the user's question. Can be replaced by keyword heuristics initially.

```
System: You are a query router. Given a user question about customer support data,
determine which data sources to search.

User: {user_question}

Respond with valid JSON only:
{
  "search_issues": true/false,
  "search_transcripts": true/false,
  "reasoning": "..."
}

Guidelines:
- search_issues: true if the question is about specific problems, bugs, features,
  patterns, or trends across customer issues
- search_transcripts: true if the question is about conversations, what agents said,
  account history, workarounds recommended, or competitive mentions
- Both can be true for broad questions like "tell me everything about X"
```

## Prompt guidelines

### Temperature settings
- Extraction (Prompt 1): `temperature=0` — deterministic structured output
- Arbitration (Prompt 2): `temperature=0` — deterministic classification
- New subtopic (Prompt 3): `temperature=0.2` — slight creativity for naming
- Centroid update (Prompt 4): `temperature=0.2` — slight creativity for synthesis
- RAG chat (Prompt 5): `temperature=0.3` — slightly creative for natural conversation

### Token budgets
- Extraction: `max_tokens=2048` (scales with transcript length)
- Arbitration: `max_tokens=256`
- New subtopic: `max_tokens=512`
- Centroid update: `max_tokens=512`
- RAG chat: `max_tokens=2048` (needs room for detailed, grounded answers)

### Error handling
All Claude calls should:
1. Wrap in try/catch with retry logic (3 attempts, exponential backoff)
2. Validate JSON output with a schema before processing (except RAG chat which returns free text)
3. Log the full prompt and response for debugging (with PII redaction)
4. Fall back gracefully: if Claude returns invalid JSON, mark the issue/task as `error` and queue for manual review
5. For RAG chat: if streaming fails mid-response, return a partial response with an error indicator
